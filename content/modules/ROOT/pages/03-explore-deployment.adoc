== Learning Objectives

By the end of this section you will be able to:

* Log into the OpenShift cluster that is running the it-self-service-agent deployment
* Identify the key workloads and understand what each one does

== The Running Architecture

You interacted with the agent in the previous step. Now look at what is actually running on OpenShift to make that work.

== Step 1: Switch to the terminal tab

A terminal tab is available in the top right of the browser window. Select the tab labeled **terminal** to switch to the terminal.

== Step 2: Log into the cluster

. Log into the cluster
+
[source,bash,role="copypaste",subs=attributes+]
----
oc login {openshift_api_url} --username {user} --password {password} --insecure-skip-tls-verify=false
----

== Step 3: Inspect the Workloads

. List the running pods
+
[source,role="execute"]
----
oc get pods
----

. Here is what each pod does:
+
[cols="1,2", options="header"]
|===
| Workload | Role

| **llamastack**
| Llama Stack server providing LLM inference and knowledge-base (RAG) APIs

| **mcp-self-service-agent-snow**
| MCP server exposing ServiceNow operations (ticket creation, querying) as tools the agent can call

| **pgvector**
| PostgreSQL database with pgvector extension; stores conversation history and provides vector search for the RAG knowledge base

| **self-service-agent-agent-service**
| Hosts the LangGraph state machine — routing agent and specialist agents live here

| **self-service-agent-db-migration**
| One-time Job that initializes the database schema; shows as Completed once done

| **self-service-agent-init**
| One-time Job that seeds initial configuration; shows as Completed once done

| **self-service-agent-integration-dispatcher**
| Receives CloudEvents from the request manager and dispatches them to the appropriate agent or external integration

| **self-service-agent-mock-eventing**
| Lightweight mock event bus used in Demo Mode in place of a full eventing infrastructure

| **self-service-agent-mock-servicenow**
| Mock ServiceNow instance used in Demo Mode so ticket creation and queries work without a real ServiceNow subscription

| **self-service-agent-request-manager**
| Receives inbound messages (email, Slack, CLI), routes them as CloudEvents, and sends replies

| **test-email-server**
| Runs three containers: the Greenmail SMTP/IMAP mail server, the test email UI you used in the previous step, and a mail polling sidecar
|===
